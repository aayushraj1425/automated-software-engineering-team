ASEP — the architecture as it exists right now

Every box is a real file or a real process on your machine. Nothing here is planned-but-not-built except where marked “later”.

1. The three processes and what's inside them

When you run pnpm dev, you start the web app and the engine. Docker runs the databases. The browser only ever talks to the web app; only the web app talks to the engine; only the engine talks to Gemini and Postgres.

2. What happens when you start an agent run (today, with stub agents)

This is the exact Day-1 flow. The “agents” pretend to work for 0.4 seconds each — everything else is real and stored in Postgres.

sequenceDiagram
    actor You
    participant Page as /runs page
    participant BFF as api/runs/route.ts
(web app)
    participant API as api/runs.py
(engine)
    participant Runner as agents/runner.py
    participant Sup as agents/supervisor.py
    participant DB as Postgres

    You->>Page: type request, click Start run
    Page->>BFF: POST /api/runs
    BFF->>BFF: check session, sign 60s pass
    BFF->>API: POST /v1/runs (with pass)
    API->>DB: insert agent_runs row (status: queued)
    API-->>Page: 201 — run id
    Page->>You: jump to /runs/{id}

    Note over Runner: continues in the background
    Runner->>DB: status → planning, event "run.started"
    Runner->>DB: insert 4 agent_tasks (the stub plan)
    Runner->>Sup: here's the task board
    loop until all tasks done
        Sup->>Sup: pick next task whose
dependencies are done
        Sup->>Runner: run this task
        Runner->>DB: task → in_progress (event)
        Runner->>Runner: stub agent "works" 0.4s
        Runner->>DB: task → done (event)
    end
    Runner->>DB: status → completed, event "run.finished"

    loop every 1.5 seconds
        Page->>BFF: GET /api/runs/{id} + /events?after=N
        BFF->>API: forward with a fresh pass
        API->>DB: read tasks + events newer than N
        API-->>Page: task board + new timeline lines
    end

3. What happens when you send a chat message

sequenceDiagram
    actor You
    participant Chat as chat-panel.tsx
    participant BFF as api/chat/route.ts
    participant API as api/chat.py (engine)
    participant Router as llm/router.py
    participant Gemini as Gemini API
    participant DB as Postgres

    You->>Chat: press Enter
    Chat->>BFF: POST /api/chat
    BFF->>BFF: check session, sign 60s pass
    BFF->>API: POST /v1/chat (with pass)
    API->>DB: load earlier messages
    API->>Router: stream("coder", history)
    Router->>Gemini: gemini-2.5-flash, stream=true
    loop each word
        Gemini-->>Router: word
        Router-->>API: word
        API-->>BFF: word (SSE)
        BFF-->>Chat: word (passed through)
        Chat-->>You: word appears
    end
    API->>DB: save both messages

4. The database tables and how they connect

Two families: better-auth owns the identity tables (created by its own migration tool); the engine owns the rest (created by Alembic migrations 0001 and 0002). They link by plain text user ids — no hard foreign keys across the families, so either tool can migrate without breaking the other.

erDiagram
    user ||--o{ session : "signs in"
    user ||--o{ conversations : "owns (by text id)"
    user ||--o{ agent_runs : "owns (by text id)"
    conversations ||--o{ messages : "has"
    repositories ||--o{ agent_runs : "runs against"
    agent_runs ||--o{ agent_tasks : "plan checklist"
    agent_runs ||--o{ agent_events : "timeline diary"
    agent_runs ||--o{ artifacts : "outputs"

    user { text id PK "better-auth" }
    session { text id PK "better-auth" }
    conversations { uuid id PK  text user_id }
    messages { uuid id PK  text role  text content }
    repositories { uuid id PK  text url }
    agent_runs { uuid id PK  text status  text request  jsonb plan }
    agent_tasks { uuid id PK  int sequence  text role  text status  jsonb depends_on }
    agent_events { bigint id PK "numbered 1,2,3… for polling"  text type  jsonb payload }
    artifacts { uuid id PK  text kind  text content }

5. Who the agents are (from engine/agents/registry.py)

flowchart TD
    SUP2["Supervisor
model: cheap tier
no editing tools — just routes tasks"]
    PM["Product Manager
model: planner tier (best model)
tools: read files, search, create tasks"]
    BE["Backend Engineer
model: coder tier
tools: read + edit files, git"]
    FE["Frontend Engineer
model: coder tier
tools: read + edit files, git"]
    OPS["DevOps Engineer
model: coder tier
tools: read + edit files, git"]
    REV["Reviewer
model: planner tier (best model)
tools: read-only + diff — cannot edit, on purpose"]

    SUP2 -->|"hands tasks to"| BE & FE & OPS
    PM -->|"writes the plan first"| SUP2
    BE & FE & OPS -->|"finished work goes to"| REV

The safety idea: agents that judge (Reviewer) or plan (Product Manager) physically have no tools that can change files. Model tiers map to .env: planner = gemini-2.5-pro, coder = gemini-2.5-flash, cheap = gemini-2.5-flash-lite.
