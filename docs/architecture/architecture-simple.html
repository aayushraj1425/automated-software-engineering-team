ASEP architecture — simple first, then explained

The simple picture (start here)

Five boxes. If you remember nothing else, remember this: requests only ever flow left to right, and answers flow back the same way. Nothing skips a box.

Website = what you see, plus the bouncer (checks you're signed in).
Engine = the brain room: talks to the AI, runs the agents, remembers everything.
Database = the memory: accounts, chats, agent runs.
AI = rented thinking, paid per word. The engine is the only box allowed to call it.

    UI -->|"① start run"| SESS
    SESS -->|"② signed in? yes"| PASS
    PASS -->|"③ 60-second pass"| RELAY
    RELAY -->|"④ forward + pass"| DOOR
    DOOR -->|"⑤ pass valid"| RAPI
    RAPI -->|"⑥ save run, reply instantly"| DB
    RAPI -->|"⑦ work continues in background"| RUN
    RUN <-->|"⑧ which task next?"| SCHED
    RUN -->|"⑨ every step written down"| DB
    WATCH -.->|"⑩ every 1.5s: anything new?"| RELAY
  1. ① Start run. You type a feature request and click Start. The page sends it to the website's own backend — never to the engine directly.
  2. ② The bouncer. lib/auth.ts (better-auth) reads your session cookie and confirms who you are. No cookie → sent to sign-in.
  3. ③ The pass-maker. lib/service-token.ts signs a small token that says “user X, valid for 60 seconds”. Both sides know the same secret (ENGINE_SERVICE_SECRET in .env), so the engine can verify it wasn't forged.
  4. ④ The relay. app/api/runs/route.ts forwards your request to the engine with that pass attached. It adds nothing else — it's deliberately boring.
  5. ⑤ The door. The engine's auth.py checks the pass on every request. Expired or fake → 401, nothing happens.
  6. ⑥ The runs desk. api/runs.py writes one row into agent_runs (status: queued) and replies to you immediately with the run's id. You're not kept waiting.
  7. ⑦ The foreman. agents/runner.py picks the run up in the background: writes the plan as rows in agent_tasks, then works through them. Today its workers are stubs; Day 2 they become real AI agents.
  8. ⑧ The scheduler. agents/supervisor.py answers one question, over and over: “which task is allowed to run next?” Rules: dependencies first, two retries max, third failure stops the run.
  9. ⑨ The diary. Every step becomes a numbered row in agent_events. This is the single source of truth for what happened — the UI is just a pretty view of this table.
  10. ⑩ Watching. The watch page asks every 1.5 seconds: “give me events newer than #N”. Because events are numbered, nothing is ever missed or shown twice — even if you refresh the page.

Chat works the same way through boxes 1–5, then goes to api/chat.py → llm/router.py → Gemini instead of the foreman — and the reply streams back word by word.
